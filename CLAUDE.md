## Repository overview

A full-stack demo application with a GitHub Actions CI/CD pipeline across three environments (dev → test → prod). The stack is:

- **Backend**: ASP.NET Core Web API (.NET 10), PostgreSQL (EF Core), Redis, RabbitMQ, SignalR, OpenTelemetry, OIDC/JWT auth
- **Frontend**: React 19 + TypeScript + React Router v7 + TanStack Query + Tailwind CSS v4, API client generated by Orval
- **E2E tests**: Playwright (`tests/playwright/`)
- **Backend integration tests**: xUnit v3 + Testcontainers (`tests/backend/WebApi.Tests/`)

## Backend commands

```bash
# Run locally (opens Swagger UI at https://localhost:7156/swagger)
cd src/backend/WebApi
dotnet run -lp "https swagger"

# Build entire backend solution
cd src/backend
dotnet build

# Run backend tests (spins up real Postgres + RabbitMQ via Testcontainers)
cd tests/backend
dotnet test

# Run a single test
dotnet test --filter "FullyQualifiedName~MyTestClass.MyTestMethod"

# Add an EF Core migration
cd src/backend/WebApi
dotnet ef migrations add <MigrationName>

# Regenerate swagger.json (happens automatically on Debug build)
ASPNETCORE_ENVIRONMENT=Swagger dotnet swagger tofile --output swagger.json bin/Debug/net10.0/WebApi.dll v1
```

## Local infrastructure

The backend requires Postgres, RabbitMQ, and Redis running locally. Start Postgres via podman:

```bash
cd src/backend
podman run --rm -e ACCEPT_EULA=Y -e SA_PASSWORD=DevelopmentOnlyPassword1 \
  -p 5432:5432 --name postgres-dev -it postgres:17
```

Default connection strings (from `appsettings.json`):
- **Postgres**: configure via user secrets (`ConnectionStrings:Blogging`) — the default value in `appsettings.json` is a placeholder
- **Redis**: `redis://:guest@127.0.0.1:6379/0`
- **RabbitMQ**: `localhost:5672`, user `guest`, password `guest`

Set the Postgres connection string via user secrets:
```bash
cd src/backend/WebApi
dotnet user-secrets set "ConnectionStrings:Blogging" "Host=localhost;Database=blogging;Username=postgres;Password=..."
```

## Architecture

### API client generation pipeline

The swagger.json at `src/backend/WebApi/swagger.json` is **auto-generated on every Debug build** by a post-build MSBuild target. Orval then reads this file and generates the TypeScript API client into `src/frontend/app/api/`. After changing backend endpoints, run `npm run orval` in the frontend to sync the client.

Orval generates:
- `app/api/endpoints/<tag>/` — one directory per Swagger tag, containing react-query hooks + MSW mock handlers
- `app/api/models/` — TypeScript types

### Authentication flow

OIDC provider: `https://auth.wsh.no`, client `pxce`. The frontend wraps the app in `AuthProvider` (`app/auth.context.tsx`), which uses `react-oidc-context`. A `TokenBridge` component inside the provider calls `setAuthToken()` on the Axios instance whenever the session changes, so all generated API calls automatically include `Authorization: Bearer <token>`.

SignalR WebSocket connections cannot carry Authorization headers; instead the JS client appends `?access_token=` to the `/hubs/chat` URL, and the backend JWT middleware picks it up specifically for paths starting with `/hubs`.

### Runtime configuration

The frontend resolves the API base URL in this priority order (see `app/public-runtime-config.ts`):
1. `window.__TEST_PUBLIC__.apiBaseUrl` (injected by the root loader's inline `<script>` at runtime)
2. `VITE_API_BASE_URL` env var (build-time, Vite)
3. `PUBLIC_API_BASE_URL` env var (runtime, Node/SSR)
4. Default: `https://localhost:7156/`

This pattern allows the same container image to target different backends per environment without a rebuild.

### Backend service layout

| Directory | Purpose |
|---|---|
| `WebApi/Controllers/` | REST controllers |
| `WebApi/Hubs/` | SignalR hubs (`ChatHub`) |
| `WebApi/Database/` | `BloggingContext` (EF Core), entity configs |
| `WebApi/Repository/` | `IBloggingRepository` / `BloggingRepository` |
| `WebApi/Messaging/` | RabbitMQ producer (`MessageSender`) and consumer (`MessageReceiver`) |
| `WebApi/Migrations/` | EF Core migration files |
| `WebApi/Options/` | Strongly-typed config classes (`OidcOptions`, `RabbitMqOptions`) |
| `WebApi/Filters/` | `ValidateModelAttribute`, `ProduceResponseTypeModelProvider` |
| `ApiClient/` | Generated C# API client (used by integration tests) |

### Frontend testing

Vitest unit tests (`app/**/*.test.{ts,tsx}`) use MSW to intercept HTTP calls. The global test setup in `test/setupTests.ts` starts an MSW server with all generated mock handlers before each test suite. Tests should use `server.use(...)` for per-test overrides.

### Backend testing

Integration tests use `Microsoft.AspNetCore.Mvc.Testing` (`WebApplicationFactory`) with real Testcontainers instances of Postgres and RabbitMQ — no mocking of the database layer.

### CI/CD pipeline

Docker images are built and pushed to the container registry on merge. Deployment is handled by Dependabot: when a new image is published, Dependabot opens a PR updating the image tag in the relevant environment manifests, promoting changes through dev → test → prod.
